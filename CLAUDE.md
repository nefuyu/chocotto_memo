# Chocotto Memo App

## プロジェクト概要
- 「位置で覚える」空間記憶を活用した、オフライン重視のシンプルメモアプリ。
- 技術スタック: Flutter, SQLite (sqflite), shared_preferences

## 設計思想
- **直感的な配置**: リスト形式だけでなく、グリッド上の好きな位置にメモを配置できる。
- **オフラインファースト**: 広告表示・課金以外はすべてオフラインで完結。バックアップ/復元は現時点で考慮しない。
- **ミニマリズム**: 文字装飾などは行わず、テキスト（と絵文字）のみを扱う。

## 開発ガイドライン
1. **TDD (Test Driven Development)**: 
   - 実装コードを書く前に、必ず `test/` フォルダ内に仕様に基づいたテストコードを作成せよ。
   - `flutter test` を実行し、テストが通ることを確認しながら実装を進めること。
2. **ブランチ運用**: 
   - 作業開始時に必ず `task/` プレフィックスの新ブランチを作成せよ。
   - 新機能・バグ修正など Issue に紐づく作業は `task/issue-[ID]` の形式とする。
   - リファクタリング・テスト修正・保守作業など Issue 不要の作業は `task/` 以下を内容に応じて自由に命名してよい（例: `task/fix-deprecated-api`）。
   - 完了後はそのブランチにコミットせよ。
3. **プライバシー**:
   - オフラインファースト。広告表示・課金に関わるSDK以外は、外部API通信やFirebase等を一切使用しない。ローカル保存のみで完結させること。
4. **Issue と実装ドキュメント (SDD)**:
   - 新機能・バグ修正は GitHub Issue を作成してから着手せよ。
   - リファクタリング・テスト修正・保守作業は Issue 作成不要。
   - Issue に紐づく作業では、実装完了後（PR作成時または実装中）に `document/` フォルダへ SDD ドキュメントを作成せよ。Issue 不要の作業では SDD ドキュメントも不要。
   - SDD ファイル名は `{YYYYMMDDHHmmss}_{機能名}.md` の形式とする（例: `20260324034522_メモ一覧表示.md`）。タイムスタンプはドキュメントを作成する時点のローカル時刻を使用する。
   - ドキュメントにはIssue概要・実装方針・クラス/関数構成・テスト方針・既知の制約を記載すること。
   - PRマージ後は実装履歴として扱い、変更してはならない。
   - ドキュメントはGit管理下に置き、コードと同じブランチにコミットせよ。
5. **Codexレビューの呼び出し**:
   - PRを作成したとき、および既存PRに新しいコミットをプッシュしたときは、そのPRに `@codex review` とコメントせよ。
   - コメントにはそれ以外の文言を含めず、`@codex review` のみを投稿すること。
6. **コミュニケーションと中断ルール**:
   - Issueの内容に矛盾がある、または仕様が不足している場合は、実装を強行しないこと。
   - 疑問点は `gh issue comment [ID] -b "質問内容"` を使ってIssueに投稿せよ。
   - 質問を投稿した後は、作業を一旦停止（待機）し、ユーザーの回答を待つこと。

## ツール使用時の説明ルール
- Bashコマンドを実行する際は、必ず `description` に日本語で何をするコマンドか簡潔に説明を入れること。
  - 例: `git status` → 「変更ファイルの確認」、`flutter test` → 「テストの実行」
- 許可を求める際にユーザーが内容を日本語で理解できるよう、説明は具体的かつ簡潔に記述すること。

## よく使うコマンド
- テスト実行: `flutter test`
- アプリ起動: `flutter run`
- 依存関係更新: `flutter pub get`
